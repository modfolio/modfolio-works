# 정공법 — 근본 수정 + 장기 시야 + 확장성

사용자가 "정공법" 이라고 말하면 아래 정의가 곧바로 활성화된다. 매번 설명하지 않아도 된다.

## 사용자 정의 (2026-04-19 세션 원문)

> "리소스 많고 시간 많으니 에러와 워닝 안 나도록 멀리 돌아가더라도 멀리 내다보고 확장성 있고, 신기술이라 하더라도 그 기술들이 업그레이드 되고 보완이 될 거라는 것에 대한 포텐셜을 감안해서 제대로 확실하게 하는 방법"

## 5 구성 원칙

1. **근본 원인 수정** — 증상 패치 금지. `@ts-ignore`, `biome-ignore`, `as any`, `--no-verify` 등 우회 금지. 타입·코드를 고친다.
2. **에러·경고 0** — 린트·타입·테스트 warning 포함 전부 통과. "나중에 정리" 로 남기지 않는다.
3. **장기 시야 + 확장성** — 가까운 해결책이 아니라 6~12 개월 뒤에도 유효한 구조 선택. 성급한 추상화는 지양하되, 명백히 재사용·확장될 경로는 선반영. **attention budget = finite resource** (Anthropic 2026, canon `attention-budget.md`) — 1M token context 도 유한. agent/skill/canon 설계 시 highest-signal token 우선, cache_control 적극 활용, multi-agent specialization.
4. **신기술 포텐셜 감안** — 현재 불완전한 도구라도 업그레이드·보완 가능성이 높으면 채택. 다만 채택 조건은 **현 시점에서 동작 + 마이그레이션 경로 명확** 두 가지.
5. **리소스·시간 투자 허용** — 돌아가는 경로가 길더라도 옳은 경로면 채택. "빠른 한 수" 로 정공법을 대체하지 않는다.

## "정공법" 트리거 시 기본 행동

- 가능한 우회·placeholder·TODO 주석 추가 금지 (실제 구현 완료까지 task 미완료)
- 수정 범위가 커져도 근본 원인까지 파고듦 (사용자 사전 허락 받은 상태)
- 신규 파일/스키마/마이그레이션 필요하면 주저하지 말고 제안
- 품질 게이트 (`bun run quality:all`) 전부 녹색 확인까지가 완료
- 검증 후 retrospective 가능한 지점 (커밋 메시지·docs·journal) 에 결정 근거 남김
- 불확실한 부분은 사용자에게 질문 (침묵 가정 금지)

## 반-패턴 (정공법 아님)

- "일단 동작만" — MVP 핑계로 엣지 케이스·에러 핸들링 누락
- `any` 로 타입 회피, 경고 억제 주석 추가
- 임시 fallback·backwards-compat shim 을 영구 방치
- "TODO: 나중에" 주석 남기고 다음 task 로 이동
- 호환성 우려로 마이그레이션 미룸 (장기 debt 누적)

## 실용 모드와의 차이

정공법 반대는 "실용 모드" — 시간·리소스 제약이 있을 때 사용자가 명시적으로 요청:

- "빨리 패치만" / "임시 방편" / "일단 우회" / "hotfix" — 이때는 최소 변경 + 나중 후속 티켓 제안
- 사용자가 명시 안 했으면 **정공법이 기본값** 이다 (이 프로젝트 CLAUDE.md 불변 원칙)

## 이 규칙의 목적

반복 설명 피로 제거. 사용자가 세션마다 "리소스 많고 시간 많으니 제대로" 라고 풀어 쓰지 않아도, "정공법" 한 단어로 위 5 원칙 + 기본 행동이 즉시 작동해야 한다.
